Security Review: Lazarus Protocol
==================================

Files Reviewed:
- core/encryption.py
- core/storage.py
- core/config.py
- cli/main.py
- cli/setup.py
- agent/heartbeat.py
- agent/alerts.py

---

CRITICAL
--------

| File | Issue |
|------|-------|
| agent/alerts.py:234 | _build_decryption_kit() never deletes tmp_dir — decryption kit temp files persist on disk indefinitely, leaking key_blob.txt |
| agent/heartbeat.py:347 | logger.critical() logs beneficiary.email (fine) but exposes config.vault.key_blob in plaintext to disk logs — the entire vault protection is bypassed if logs are compromised |
| core/encryption.py:193 | del aes_key is ineffective — Python doesn't guarantee memory zeroing; the key remains in memory indefinitely |

---

HIGH
----

| File | Issue |
|------|-------|
| agent/alerts.py:246-328 | Embedded decrypt.py has no input validation — paths aren't checked for traversal (../../etc/passwd), private key is loaded with backend=default_backend() deprecated, no error handling for corrupted keys |
| agent/heartbeat.py:366-368 | If send_delivery_email raises after logging the trigger, disarming never runs — agent will re-trigger on next heartbeat, potentially re-sending emails |
| agent/heartbeat.py:394 | watchdog_job re-registers heartbeat_job() without config_path — recovery fails if agent was started with a non-default config path |
| core/config.py:186 | chmod 600 only runs on POSIX — on Windows the key_blob-containing config file has no access controls |

---

MEDIUM
------

| File | Issue |
|------|-------|
| core/encryption.py:98 | BestAvailableEncryption uses unspecified PBKDF2 iterations — not future-proof; should use kdf=PBKDF2 with explicit iteration count |
| core/storage.py:147 | data.get("cid", {}).get("/") silently falls through if cid key exists but is falsy — could return malformed CID |
| core/storage.py:231-244 | No file size limit on IPFS downloads — malicious large file could exhaust disk/memory |
| cli/setup.py:93 | Returns dict instead of BeneficiaryConfig — type inconsistency with _config_from_dict() which expects dataclass fields |
| agent/alerts.py:306-309 | decrypt.py assumes key_blob.txt is in current working directory — will crash if user unzips elsewhere and runs from different dir |

---

LOW
---

| File | Issue |
|------|-------|
| core/config.py:237 | if since == float("inf") uses exact float equality — precision edge cases could bypass infinity check |
| core/config.py:41 | VAULT_PATH defined but unused — confusing dead code |
| core/storage.py:40-41 | Local IPFS gateway defaults to http://127.0.0.1:8080 — no HTTPS enforcement if user overrides to non-localhost |
| agent/heartbeat.py:79 | PID file written without permissions — another local user could overwrite/corrupt it |
| cli/setup.py | All functions are NotImplementedError — no security review possible yet, but email format validation is missing |

---

Summary
-------

1. Temp file leak (alerts.py) is the most immediate data-leak risk
2. Log exposure of key_blob (heartbeat.py) completely defeats the encryption scheme if logs are stored or shipped
3. Input validation in embedded decrypt.py is critical for the inheritance use case — a non-technical beneficiary could be exploited
4. Windows has no file permission protection for the key_blob
